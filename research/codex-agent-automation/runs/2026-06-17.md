## New Facts

- Codex automations support recurring background tasks and thread heartbeats
  for workflows that should preserve conversation context. Source:
  [Codex Automations](https://developers.openai.com/codex/app/automations).
- Project-scoped Codex automations require the local Codex app machine to be
  powered on, Codex to be running, and the selected project to remain available
  on disk when the schedule fires. Source:
  [Codex Automations](https://developers.openai.com/codex/app/automations).
- In Git repositories, automations can run in the local project or a dedicated
  worktree. Worktrees isolate automation changes from unfinished local work;
  local mode can modify files currently being edited. Source:
  [Codex Automations](https://developers.openai.com/codex/app/automations).
- Automations use default sandbox settings. Read-only mode blocks writes,
  network access, and app control; full access carries elevated unattended-risk;
  and automations use `approval_policy = "never"` when organization policy
  allows. Source:
  [Codex Automations](https://developers.openai.com/codex/app/automations).
- Codex skills are the right reusable workflow layer for this MVP because they
  package instructions and optional resources that automations can invoke.
  Source: [Codex Skills](https://developers.openai.com/codex/skills).
- MCP is the integration layer for live tools and external context. Source:
  [Codex MCP](https://developers.openai.com/codex/mcp).
- Hooks are suitable for later enforcement and logging, but they are not needed
  for the MVP daily research loop. Source:
  [Codex Hooks](https://developers.openai.com/codex/hooks).
- The Codex SDK is the most direct V2 path for a VPS/server orchestrator.
  Source: [Codex SDK](https://developers.openai.com/codex/sdk).
- The app-server docs explicitly position app-server for rich client
  integrations and point automation or CI jobs to the Codex SDK instead. Source:
  [Codex App Server](https://developers.openai.com/codex/app-server).
- `codex exec` defaults to a read-only sandbox and should use explicit sandbox
  settings for automation. Source:
  [Codex Non-interactive Mode](https://developers.openai.com/codex/noninteractive).
- The official Codex manual helper reported the local manual cache was current
  after network approval. Before approval, the same helper failed with DNS
  resolution for `developers.openai.com` in the sandboxed command context.
  Source: current-session command output and
  [Codex Automations](https://developers.openai.com/codex/app/automations).
- Local runtime check reports `codex-cli 0.140.0-alpha.2` in this workspace.
  Source: current-session `codex --version` output.

## Risks

- Daily research can become noisy if topic contracts are too broad.
- Unattended write access needs a strict risk gate.
- Project-scoped automations can miss runs if the local machine is off, Codex is
  not running, or the project path disappears.
- Running automations in local project mode can modify files that are also being
  edited manually.
- Product behavior can change, so current Codex facts should be verified
  against official docs when making implementation decisions.
- If future unattended runs cannot refresh official docs because network access
  requires approval, they may need to use the last cached manual and clearly
  report that limitation.
